因为面的senior，第一轮就是sd，题目是地里那道design file system
注意：
和alex xu的s3那本书不同，题目要求listing的操作要和put/modify file consistent，所以不能用那本书的table schema，不是object，而是directory，我认为应该是用directory 作为dynamodb的partition key，可以用subdir或者文件名做sorting key，也可以像那本书里一样用某种fileId作为sortingkey，这样可以允许同样file多个version，否则，如何一边更新file一边下载file是个问题。
另外如果directory做为partitionkey的话，可以用localsecondaryindex，（自然做到consistent），可以很容易查找所有subdir或者所有文件（引入一个fileType field create LSI即可）
被问到如何consistent remove direcotry，难点就是如何remove很多subdirectory，似乎可以把每个subdir下面的subdir放到一个deque中，multiple thread来pull，每个thread就是mark db table一个field标志这个subdir 已经deleted。
data store那部分我按照hdfs/gfs答的

一些补充（地里真bt居然不允许直接编辑）1. 面试官问了如果文件太大怎么办，当然需要引入一个chunkTable，这时就可以看出额外的fileId有帮助，不但可以解决multiple version的问题，而且fileId ChunkNum就只需要维护两个field。2 我自己脑抽聊了一下如果文件太小怎么办。切记切记alex那本书有点问题，其实小文件影响不大，不需要像书中那样把小文件放一起，inode很难exhausted，而且如果小文件实在太多可以加storage 机器。3 如果没有准备好，不要聊hdfs如何保证namenode 如何克服single point failure，我先前看一些文档说需要zookeeper/ztcd，但是其实很简单的active/standby就行。 我面挂的原因极有可能是因为我讲了些面试官不care的东西。
