DB的系统设计分为传统的high level system design(architect design)和low level system design. 第一种就是常见的画画high level workflow的那种. 第二种low level system design比较特殊, 需要写出伪代码, 并且最好可以run. 感觉更像学生时期的那种cs课的小project, 但需要在很短的时间之内把所有东西想通并且写出来. DB题库感觉比较小, 很多时候都能遇到原题, 所以提前多准备一下会很有帮助.

Architecture(High Level System Design)
Bookstore
problem
用户发一个max price，遍历dependency几百个书店api找到最便宜价格，返回给用户
设计一个卖书的web service，顾客发来书，信用卡以及最高接受的价格，我们的service拿着这个价格去后台连接的几百个bookstore去找这本书的最低价格，如果最低价格低于顾客给的最高接受价格那就成交付款，否则返回这个最低价格给顾客。
真的就是这么简单。latency要求10-20 seconds，book seller有50-200个，book有1-2 million，scale非常小。

问题描述
Serving infra for both internal and external users, with data flow like Client -> HTTP server (Gateway) -> API server -> database / 3P server / Another API server API servers are owned by internal teams (1P) There's a burst of traffic. 股票交易系统
req
例如客户可以发送以下请求： BUY BABA 30 “2023-11-01 15:00:00” SELL AMZN 10 “2023-11-01 12:00:00” broker 提供以3个接口：
处理客户发送的股票交易请求，该系统需要对接broker提供的api，在截止时间前买入或卖出某公司一定数量的股票。
如果到了截止时间后交易没有成功，则需要取消该交易。（这里不需要关心交易能否成功，调用broker接口会告诉我们状态
某个GET接口，用于获取订单交易状态
某个POST接口，用于创建股票交易订单
某个DELETE接口，用于取消某笔交易
也有scheduling的考察. 决定一个order的state 需要在deadline的时候进行check。 说明需要一个scheduler在这个stock id 在这个timeline check broker. 而且可能ddl这个点刚好有大量order需要check，该如何优化
