## Post

数据砖5轮高强度轰炸
Coding + Algo: 这两个其实是一样的，题库是shared，之前疑惑了半天有什么区别。
第一轮 Coding考了tiktactoe，player是自动依次play，eg：第一次call play() function是player 1， 第二次call player() function就默认player 2. 需要打出每一步下棋之后的棋盘的样子，如果赢了就输出“Player X won”，要求有格式，然后自己写test case，面试官给了十几个test case，一一run完。然后问了follow up问题是设计一个AI player 让AI random的下棋，直到游戏结束。中途忘了generate random number怎么写了，于是现场google怎么写，一个bug是tictactoe 有可能平局的，尤其AI play的时候，平局需要结束游戏，然后和小哥聊了会儿天，结束。
Algo是KV store + hit counter，就是写一个key value store，然后求averageGet，averagePut，这题没有给example code，因此function，parameters全部都得自己定义，得问清楚面试官想要啥。 网上目前有很多这个题的乱七八糟的参考，方法1.做一个queue，方法2. 一个array记录total，然后二分法找出hit counts。 方法3. bucket， 并且dynamic buckets。 我完全没看方法2，然后说了方法1，面试官说不够efficient， 然后说方法3，说不够dynamic。。。 然后楼主就g在当场了，最后面试官只能说让我写方法3，并且自己写test cases，然后我写get/put的functionality的时候是void put(string key, string value, int timestamp),面试官问如何去掉parameter timestamp然后测试，我说那就只能call current system time？ 但这个写unit test可能不够方便，面试官说不是，我说不然默认每一个put/get都间隔一秒？ 面试官说不行，然后最后也没搞清楚面试官想要啥。。。
Arch
Bookstore Platform, 面试官问如果就30个vendors，然后每秒几百个call怎么做，然后如果用户找到了在价格范围内的书就自动下单，无需再下一个request， 然后我提出了为了避免oversell每次payment的时候加lock，面试官表示会影响performance，为什么加lock，我说如果一些vendor在concurrent purchase的时候有可能出现race condition/oversell的情况，面试官说可以return payment failure，问我你依然认为需要加lock吗？ 还有payment大概擦边问了一下为什么选择store payment_token， 然后如果只有30个vendor，其实很多大招都用不上。。。 还有一些很细节的问题，刨根问底，而且还会考虑一些user behavior上的optimization，总之就是光showoffer上那些事不太够的要灵活使用
