## Post

前两天面的 设计一个可以存file的system 支持 uploadfile（），viewfile（）, CRUD diretory. 和list directory
今天收到拒信 觉得自己答的还ok 确实有flaw 大概思路如下
- metadata 和 file content分开
- client用来把大file分成chunk 一个一个小chunk来处理
- redis用来加速读 key file_uuid or directory_uuid， value是path for file， set（file_uuid）for direcotry
- 【错误】metadata用来组装file then send it back to client，应该让client去组装这样可以节省带宽。这个是自己的低级错误
- recursive的create和delete， 比如 a/b/c.txt，就先create a， 然后create a/b 然后a/b/c.text， 同时更新redis。 delete direcotry同理, 找到所有的这个direcotry下面的sub dir和file， 锁table， remove， 释放table
- async remove file handle in chunk service 因为file reorg很费事费力 所以用message queue来处理 当metadata service更新后立即返回remove 成功
- 用write ahead log（sequential write）+ cron job来做file replica
过程中有几个点我和面试官有分歧
1。我不认为strong consistency是一个必须的non functional requirement 他对此好像很不认同， turn out这个系统如果是只读的话根本不太可能就有consistency的问题。 我说咱们后面可以在c 和 p中取舍
2. 我说具体的replica我希望留给client自己来configure，比如备份几个 备份的strategy 我们来create lie of service 他好像也不太认同
总之过程中没收到太多有用的东西 今天通知挂了
