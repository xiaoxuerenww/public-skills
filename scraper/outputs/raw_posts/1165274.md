解决方案：最大后缀匹配（Maximal Suffix Matching / Alignment）
我们需要改变算法逻辑，不再寻找“公共前缀”，而是寻找两个列表之间的最佳重叠部分，并以此作为锚点来推断发生了什么。
算法逻辑
寻找重叠（Alignment）： 对比 OldStack 和 NewStack，找到它们共享的最长连续子序列。通常这个子序列会包含栈底（Leaf）或接近栈底的部分。
推断变化：
重叠部分右侧（Leaf 侧）：
NewStack 比重叠部分多出来的 -> Start (Push)
OldStack 比重叠部分多出来的 -> End (Pop)
重叠部分左侧（Root 侧）：
这部分的变化通常是因为窗口滑动（Shift）导致的。
关键原则： 左侧消失的帧不要生成 End 事件（因为它们还在运行，只是被移出了视野）。左侧新增的帧不要生成 Start 事件（因为它们早就开始运行了，只是刚进入视野）。

这是一个非常棒的进阶问题！在实际的性能分析（Profiling）中，为了节省带宽或内存，采样器经常只记录Call Stack 的最后 N 层（例如只记录最近的 5 个函数调用）。
在这种情况下，你丢失了绝对的“栈深度”和“根节点（Root）”信息。传统的**从根部对齐（Prefix Matching）**算法会失效，因为你无法确定两个样本的基底是否相同。
核心挑战：窗口滑动（Window Sliding）
假设你的记录窗口大小 N=3，而真实的调用栈非常深：
T=1.0: 真实栈 A -> B -> C -> D。你看到的样本：
T=2.0: 函数 E 被调用（Push）。真实栈 A -> B -> C -> D -> E。你看到的样本：[C, D, E]
如果你直接对比这两个列表：
旧：
新：[C, D, E]
如果从左向右（Index 0 vs Index 0）对比：B != C。传统算法会误以为整个栈都换了，产生错误的“B结束，C开始...”事件。但实际上，C 和 D 并没有变，只是它们在窗口中的位置移动了。
