从 Amazon SDE 转 SDM 的头三个月：几件没人提前告诉我的事
. From 1point 3acres bbs
大概三个月前我在 Amazon 从 SDE 转成了 SDM（第一次带人），想趁记忆还新把真正让我意外的几件事记下来。一方面给自己梳理，另一方面当初准备转管理时，网上能搜到的要么是"要有同理心"这类正确的废话，要么是没有 takeaway 的吐槽贴，所以想写点具体的。

简单背景：之前做了几年 SDE，是组里遇到硬 bug 和 design review 大家会来找的那个人，后来在自己组里转成了 SDM，组不大。下面是我做错、然后慢慢想明白的几件事。

1）你的产出不再是"你的"产出。
头三周我很难受，因为按我以前的标准我"什么都没做出来"——没有 commit，没有挂着我名字的 narrative 或 design doc。过了好一阵才真正接受：我的产出现在是整个组的产出，一个 20 分钟、能帮别人解开 block 的对话，比我自己写那个 PR 更值钱。衡量标准变了，但本能没跟上。
. Waral dи,
2）1:1 才是真正的工作，不是"额外开销"。
我以前把 1:1 当成对进度的会。错了，进度在 ticket 里。1:1 是唯一能稳定拿到那些不会出现在 standup 里的信息的渠道——某人在默默 burnout、某人觉得 promo 卡住了、两个队友之间有摩擦。我后来把它当成"他们的时间"而不是我的：议程他们定，我主要问问题然后闭嘴。拿到的信息质量直接上了一个台阶。

3）最难的技能是"忍住不 dive in"。. check 1point3acres for more.
Amazon 把"Dive Deep"在你做 IC 的那些年里刻进了骨子里，所以看到队友卡在一个我十分钟就能修好的问题上，我浑身本能都在喊"直接接过来"。但每次这么做，我都是在教他"反正经理会来救我"，同时抢走了他练手的机会。现在我会先问"你试过什么 / 你的计划是什么"，然后把手揣兜里。说实话很难受，但这恰恰是重点。

一些更小但有用的习惯：
- feedback 当天发现当天给，不攒到 Forte 季。小、具体、快。攒到年度 review 再一次性倒给人，等于偷袭。
- 每个 report 开一个 running doc 记 wins 和 concerns，写 review 时就不用靠脑子硬回忆。
- 花真正的精力在 manage up 和"翻译"上——把组里的工作翻译成 leadership 在乎的语言，再把 leadership 那边的混乱翻译成组里能落地的稳定计划。帮组里挡住上面的折腾，是这份工作的一半。. From 1point 3acres bbs
- 有意识地放下"我是这屋里最强的工程师"这个身份。是它把你带到这里的，但抱着不放只会让你变成更差的经理。

还在学习中。如果你是有经验的 SDM，特别想听听你"希望自己第三个月就知道"的事。如果你是还在犹豫要不要转管理的 SDE，评论区随便问，我尽量答。
